5 # Features {#qtrestclient_readme_label_0}
6 - Consume any JSON-REST-API
7 - Map API objects to QObject/Q_GADGET classes. Supports:
8  - basic objects
9  - lists
10  - paging objects
11 - Allows to create API class representations to wrap specific parts of the api
12 - Reply-based system - use either signal/slot or a more functional approach
13 - Custom compiler to generate API objects, classes and methods from simple JSON files

38 # Usage {#qtrestclient_readme_label_2}
39 The restclient is provided as a Qt module. Thus, all you have to do is add the module, and then, in your project, add `QT += restclient` to your `.pro` file!
40 
41 The API consists of 3 main classes:
42 - **RestClient:** Set up the API, i.e. how to access it, the base URL and headers
43 - **RestClass:** A subset of the API, allows to make requests
44 - **RestReply:** The reply control returned for every request, to obtain the result and react on errors
45 
46 ## Example {#qtrestclient_readme_label_3}
47 The following example shows an example request made to [JSONPlaceholder](https://jsonplaceholder.typicode.com/).
48 
49 ### API-Definition {#qtrestclient_readme_label_4}
50 Before we can start using the API, we have to define c++ objects to map the JSON to. For this example, all we need is to represent a post. This example makes use of Q_GADGET, but it works the same for QObjets as well. MEMBER properties are used for simplicity, it works with any normal property. See [QtJsonSerializer](https://github.com/Skycoder42/QJsonSerializer) for more details about the serialization part. **Note:** This is optional. You can in fact use the API with JSON only.
51 ```cpp
52 struct Post
53 {
54  Q_GADGET
55 
56  Q_PROPERTY(int id MEMBER id)
57  Q_PROPERTY(int userId MEMBER userId)
58  Q_PROPERTY(QString title MEMBER title)
59  Q_PROPERTY(QString body MEMBER body)
60 
61 public:
62  Post();
63 
64  int id;
65  int userId;
66  QString title;
67  QString body;
68 };
69 ```
70 
71 ### API-Use {#qtrestclient_readme_label_5}
72 We will now query the data of the post with the id 42. The final URL is: https://jsonplaceholder.typicode.com/posts/42
73 
74 First, you set up the client, for example in your main:
75 ```cpp
76 auto client = new QtRestClient::RestClient();
77 client->setBaseUrl(QUrl("https://jsonplaceholder.typicode.com"));
78 QtRestClient::addGlobalApi("jsonplaceholder", client);//optional, makes it easier to use the api
79 ```
80 
81 Now, in any class you want to use the api, you can create rest classes to do so:
82 ```cpp
83 auto restClass = QtRestClient::createApiClass("jsonplaceholder", "posts", this);//creates a restclass for the registered API, with the sub-path "posts"
84 ```
85 
86 Finally, we send the get-request and log the reply:
87 ```cpp
88 restClass->get<Post>("42")->onSucceeded([](RestReply *reply, int statusCode, Post data) { //calls: GET https://jsonplaceholder.typicode.com/posts/42
89  //do whatever you want with the data
90  qDebug() << data.id
91  << data.userId
92  << data.title
93  << data.body
94 });
95 ```
96 
97 And thats all you need for a basic API access. Check the documentation for more and *important* details about the client, the class and the replies!

99 ## API-Generator {#qtrestclient_readme_label_6}
100 The library comes with a tool to create API data classes and wrapper classes over the rest client. Those are generated from JSON files and allow an easy creation of APIs in your application. The tool is build as a custom compiler and added to qmake. To use the tool simply add the json files to you pro file, and the sources will be automatically generated and compiled into your application!
101 
102 ### Example {#qtrestclient_readme_label_7}
103 The following example shows a JSON file to generate the post type from above (with shared data optimizations, automatic registration, etc.). It could be named `post.json`:
104 ```json
105 {
106  "$type": "gadget",
107  "$name": "Post",
108 
109  "id": "int",
110  "userId": "int",
111  "title": "QString",
112  "body": "QString"
113 }
114 ```
115 
116 A definition for a very simple API, that allows some simple operations for posts, could look like this. It could be named `api.json`:
117 ```json
118 {
119  "type": "api",
120  "name": "ExampleApi",
121  "includes": ["api_posts.h"],
122  "globalName": "jsonplaceholder",
123  "baseUrl": "https://jsonplaceholder.typicode.com",
124 
125  "classes": {
126  "posts": "PostClass"
127  }
128 }
129 ```
130 
131 You can either directly add the methods to the API, or, like I did in this example, split the classes into different files. It could be named `api_posts.json`:
132 ```json
133 {
134  "type": "class",
135  "name": "PostClass",
136  "includes": ["post.h"],
137  "path": "posts",
138 
139  "methods": {
140  "listPosts": {
141  "verb": "GET",
142  "returns": "QList<Post>"
143  },
144  "post": {
145  "verb": "GET",
146  "pathParams": ["id,int"],
147  "returns": "Post"
148  },
149  "savePost": {
150  "verb": "POST",
151  "body": "Post",
152  "returns": "Post"
153  },
154  "updatePost": {
155  "verb": "PUT",
156  "pathParams": ["id,int"],
157  "body": "Post",
158  "returns": "Post"
159  },
160  "deletePost": {
161  "verb": "DELETE",
162  "pathParams": ["id,int"],
163  "returns": "void"
164  }
165  }
166 }
167 
168 ```
169 
170 These files are added to the `.pro` file as follows:
171 ```pro
172 # Load the feature
173 load(qrestbuilder)
174 
175 REST_API_OBJECTS += post.json
176 REST_API_CLASSES += api.json \
177  api_posts.json
178 ```
179 
180 And thats it! once you run qmake and compile your application, those APIs will be generated for you. The usage is fairly simple:
181 ```cpp
182 //by using the API directly:
183 auto api = new ExampleApi(this);
184 auto reply = api->posts()->post(42);
185 //continue as usual with any generic network reply
186 
187 //by using the factory to create the class only:
188 auto posts = ExampleApi::factory().posts().instance(this);
189 auto reply = posts->post(42);
190 ```
191 
192 Check the JsonPlaceholderApi example in `examples/restclient` for this example.

194 ## Authentication {#qtrestclient_readme_label_8}
195 When I was trying to implement authentication, I found that it can be quite heterogenous. Because of that, I did not implement a specific authentication mechanism. Instead, you can either use one supported by Qt or create your own auth flow. The easiest way to do so is to create a RestClient, and then perform whatever authentication you need with the clients internal QNetworkAccessManager, and set parameters, headers, cookies, etc accordingly, if neccessary.
196 
197 Qt itself supports the following kinds of authentication:
198 - [**QAuthenticator:**](https://doc.qt.io/qt-5/qauthenticator.html) Integrated in QNetworkAccessManager, via a signal. See Qt Docs for supported mechanisms
199 - [**QtNetworkAuth:**](https://doc.qt.io/qt-5/qtnetworkauth-index.html) A Technology preview providing OAuth and OAuth2 authentication flows.
200 
201 **Note:** For the latter, I may add support in a Future version. For the first one, simply connect to the managers signal.

203 ## QObject Ownership {#qtrestclient_readme_label_9}
204 If you are using QtRestClient with QObjects, please be aware that none of the RestClients functions take ownership of the returend objects. This means **you** are responsible for deleting them, if not needed anymore. In short, you as caller own the objects returned to your handlers. If you are uncertain whether you need to handle an object or not, check the documentation of the specific function for details.
